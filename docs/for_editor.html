Editor‘s documentation

This documentation explains the backend work needed to create ParsedClassics collections.

To understand adequately Editor‘s documentation you should first study the Reader‘s documentation.

Editorial roles

There are two editorial roles: senior editor and resource editor. Senior editor must know basics of HTML and Javascript coding.

Senior editor

Senior editor performs the following tasks:

- defines edition;
- defines collection and its resources;
- sends the files to work with to resource editors;
- explains tools to be used and formatting to resource editors;
- checks formatting of the files created by resource editors;
- uploads created files to the server.

Resource editor

Resource editor performs the following task:

- creates resource files.

Editor‘s main tools

All editing work is being done on the localhost. The main tools are coding editor to create and edit files and web browser to view the result.

In order to start to work on ParsedClassics project, the editor should place the copy of ParsedClassics app into some directory on localhost, open /app.html file in web browser and fire the coding editor program. The editing work consists in making edits in the code editor, saving and previewing results in the web browser.

Coding editor software

Any HTML/CSS/Javascript coding editor software is suitable for ParsedClassics projects. We recommend VS Code (Win/Mac, free).

Greek polytonic keyboard

To type Greek characters Greek polytonic keyboard software should be used. It is recommended to use virtual Greek polytonic keyboard, because it implements encoding conventions required for resources of ParsedClassics collections.

Web browser

Any major modern webbrowser – Chrome, Firefox, Opera, Safari – is suitable for previewing ParsedClassics projects on localhost. Among them Firefox has certain advantage: it preserves form values while reloading the page. But in Firefox default security settings may prevent the webfonts to be rendered in localhost environment. Defining edition

To define a new edition senior editor should add JSON object into the file /data/_catalogues/_sets.js . The order of JSON objects in this file is important because Catalogue page displays the list of editions in the same order.

For example, to define the edition "Ἡ Καινὴ Διαθήκη. Πράξεις, Ἐπιστολαί, Ἀποκάλυψις / The New Testament. Acts, Epistles, Revelation" such JSON object has been added:

  nt_apostolics: {
    title_orig: 'Ἡ Καινὴ Διαθήκη. Πράξεις, Ἐπιστολαί, Ἀποκάλυψις',
    title_orig_short: 'Κ.Δ. Πράξεις, Ἐπιστολαί, Ἀποκάλυψις',
    title_eng: 'The New Testament. Acts, Epistles, Revelation',	
    collections: [],	
  },

The value of the key "title_orig" should be a string in original language. It should include author or scripture name and the title. Abbreviation or/and shortened title can be used in order to avoid very long value. This value will be used in Catalogue page.

The value of the key "title_orig_short" should be a string in original language. This value usually is shorter than the value of the key "title_orig". Abbreviation or/and shortened title can be used in order to avoid very long value. This value will be used inside Collections selectbox at Library app page.

The value of the key "title_eng" should be a string in English. It should include author or scripture name and the title. Abbreviation or/and shortened title can be used in order to avoid very long value. This value will be used in Catalogue page.

The value of the key "collections" should be array of strings which are shortnames of collections. For now let's leave this array empty; we will add member to this array after defining certain collection.

After editing /data/_catalogues/_sets.js file refresh Catalogue page. A new value should appear in editions list.

Defining collection and its resources

Defining collection and its resources consists of the following tasks:
(1) listing collection;
(2) registering collection as part of edition;
(3) creating empty collection catalogue;
(4) listing collection resources;
(5) listing scanned books.

Listing collection

To list a new collection senior editor should add JSON object into the file /data/_catalogues/_collections.js as the value of certain key of ParsedClassicsCollDefs object.

For example, to list the collection Epistle to Titus such JSON object has been added:

const ParsedClassicsCollDefs = {
  
  // some lines have been omitted

  nt_titus: {
    author_orig: 'Ἡ Καινὴ Διαθήκη',   
    author_eng: 'The New Testament',
    collection_selectboxname: 'Κ.Δ. Πρὸς Τίτον',
    collections_page_title_orig: 'Παύλου τοῦ ἀποστόλου ἡ πρὸς Τίτον ἐπιστολή',
    collections_page_title_eng: 'Apostle Paul\'s epistle to Titus', 
  },
  
  // some lines have been omitted

}

The key "nt_titus" is collection's shortname of the collection being added. Collection's shortname should be a string containing only lowercase latin letters, numbers and underscores (no spaces!). It should include author or scripture name and the title. Abbreviation or/and shortened title can be used in order to avoid very long value.

The value of the key "collection_selectboxname" should be a string in original language. It should include author or scripture name and the title. Abbreviation or/and shortened title can be used in order to avoid very long value. It will be displayed in Collections selectbox.

The key "collections_page_title_eng" should have as its value the title of the collection in English.

The key "collections_page_title_orig" should have as its value the title of the collection in original language.

Both English and original title will be displayed in Collections page. Original title will also be displayed as title of parsed text resource.

The key "author_orig" should have as its value the name of the author or the name of the scripture in original language, i. e. in Greek or Latin. It will be displayed as author name of parsed text resource.

The key "author_eng" should have as its value English name of the author or English name of the scripture. The value is currently not used.

Since classics text usually consists of several books and each book usually will have its own collection, the values of the keys "author_orig" and "author_eng" will be repeated listing those collections. In order to avoid repetitions it is recommended to define JSON object containing all keys whose values would be repeated. That JSON object should be the value of certain key of ParsedClassicsCollProtos object which is in the same file as ParsedClassicsCollDefs object. Since the values of the keys "author_orig" and "author_eng" will be repeated for all NT books, let's name the key "nt_book":

  const ParsedClassicsCollProtos = {

    // some lines have been omitted

    nt_book: {
      author_orig: 'Ἡ Καινὴ Διαθήκη',   
      author_eng: 'The New Testament',
    },

    // some lines have been omitted
  
  };

Now we can destructure the object ParsedClassicsCollProtos.nt_book anywhere we need the same values of "author_orig" and "author_eng" keys:

  const ParsedClassicsCollDefs = {
    
    // some lines have been omitted
  
    nt_titus: {
      ...ParsedClassicsCollProtos.nt_book,
      collection_selectboxname: 'Κ.Δ. Πρὸς Τίτον',
      collections_page_title_orig: 'Παύλου τοῦ ἀποστόλου ἡ πρὸς Τίτον ἐπιστολή',
      collections_page_title_eng: 'Apostle Paul\'s epistle to Titus', 
    },
    
    // some lines have been omitted
  
  }

Registering collection

The collection we just listed among other collections should be registered as a part of edition. This should be done by entering collection's shortname as member of array which is value of the key "collections" in edition's definition:

  nt_apostolics: {
    title_orig: 'Ἡ Καινὴ Διαθήκη. Πράξεις, Ἐπιστολαί, Ἀποκάλυψις',
    title_orig_short: 'Κ.Δ. Πράξεις, Ἐπιστολαί, Ἀποκάλυψις',
    title_eng: 'The New Testament. Acts, Epistles, Revelation',	
    collections: [
      'nt_titus',
    ],	
  },

The order of collection shortnames is important, because list of collections will be displayed in that order.

Listing collection

To list a new collection senior editor should add JSON object into the file /data/catalogues/_collections.js . The order of JSON objects in this file is important because collections selectbox displays the list of collections in the same order.

For example, to register Epistle to Titus collection such JSON object has been added:

, {
  
    "collection_shortname": "nt_titus"

	, "collection_selectboxname": "Κ.Δ. Πρὸς Τίτον"
	
	, "collection_set": "nt_apostolics"
  
 }

The value of the key "collection_shortname" should be a string containing only lowercase latin letters, numbers and underscores (no spaces!). It should include author or scripture name and the title. Abbreviation or/and shortened title can be used in order to avoid very long value.

The value of the key "collection_selectboxname" should be a string in original language. It should include author or scripture name and the title. Abbreviation or/and shortened title can be used in order to avoid very long value.

The value of the key "collection_set" should be a key from JSON object in the file /data/catalogues/_sets.js, which contains definitions of the editions. By entering the key "nt_apostolics" as the value of the key "collection_set" we are making Epistle to Titus collection a part of the edition "Ἡ Καινὴ Διαθήκη. Πράξεις, Ἐπιστολαί, Ἀποκάλυψις / The New Testament. Acts, Epistles, Revelation", whose shortname is "nt_apostolics".

After editing /data/catalogues/collections.js file refresh Library app. After that a new value should apper in collections selectbox. If you select this new value, both Parsed text pane and Resources pane should be empty.

Listing scanned books

To list a new scanned book senior editor should add a key and a JSON object as its value into the file /data/catalogues/_scanned_books.js . The order of JSON objects in this file is not important.

For example, to list E. Robinson‘s book Greek and English lexicon of the New Testament the key "greek_and_english_lexicon_of_nt_by_robinson" and JSON object as its value has been added:

, "greek_and_english_lexicon_of_nt_by_robinson": {
		
"bookTitle": "A Greek and English lexicon of the New Testament."

, "bookUrlText": 'By Edward Robinson. Second edition. 1850. New York: Harper and Brothers.'

, "numLeafs": "816"

, "thumbnail": "page0001.jpg"

, "metadata": [

{

"label": "Title"

, "value": "A Greek and English lexicon of the New Testament. By Edward Robinson. Second edition. 1850. New York: Harper and Brothers."

}

]
		
}

As you see the key "greek_and_english_lexicon_of_nt_by_robinson" includes at the beginning lowercase book title whose words are separated by underscores, the string "_by_", and lowercase author surname at the end.

This key is also used as directory name: /_scans/ greek_and_english_lexicon_of_nt_by_robinson . This directory contains images of the pages of the scanned book.

The JSON object‘s key "bookTitle" should have as its value scanned book title in original language.

The key "bookUrlText" should have as its value all remaining bibliographic data of the book (author, edition, year, place, publishing house).

The key "numLeafs" should have as its value number of files in the directory containing images of the pages.

The key "thumbnail" should have as its value the filename of the image displaying the title page of the scanned book.

The key "label" should always have as its value the word "Title".

The key "value" should have as its value the string containing all relevant bibliographic data of the scanned book. Usually you will get it by combining values of the keys "bookTitle" and "bookUrlText".

Creating empty collection catalogue

Collection catalogue is a file inside /data/catalogues directory. Collection catalogue‘s filename should be the same as the value of the key "collection_shortname" using which collection has been registered. In the example above we registered collection using string "nt_titus" as the value of such key. That means the name of the file of the collection catalogue should be "nt_titus.js".

The encoding of the collection catalogue file must be UTF-8 (no BOM).

Let‘s as example to create catalogue for Epistle to Titus collection:

ParsedClassicsData.loadCollectionData({

  "collection_shortname": "nt_titus"

  , "author_eng": "The New Testament"

  , "author_orig": "Ἡ Καινὴ Διαθήκη"
  
  , "collections_page_title_eng": "Epistle to Titus" 
  
  , "collections_page_title_orig": "Ἡ πρὸς Τίτον ἐπιστολή" 
  
  , "contents_shortname": "coll_nt_titus_contents"

  , "enabled_resources": []
  
  , "parsed_text": ""

  , "original_texts": []
	
  , "concordance": []
  	
  , "lexicons": []
  
  , "translations": []
  	
  , "commentaries": []
  
  , "grammar_refs": []
  	
  , "diagrams": []
    
  , "audio": []

});

JSON object‘s key "collection_shortname" should have as its value the same string which was used for listing collection.

The key "author_eng" should have as its value English name of the author or English name of the scripture.

The key "author_orig" should have as its value the name of the author or the name of the scripture in original language, i. e. in Greek or Latin.

The key "collections_page_title_eng" should have as its value the title of the collection in English.

The key "collections_page_title_orig" should have as its value the title of the collection in original language. Both English and original title will be displayed in Collections page.

The key "contents_shortname" should have as its value the name of the file containing collections contents. The name of this file is usually formed by adding the string "coll_" at the beginning and the string "_contents" at the end of the collection shortname. So, if the collection shortname was "nt_titus", the the name of collection contents file will be "coll_nt_titus_contents". Collection contents files are placed in /data/contents directory.

The key "enabled_resources" should have as its value array of the shortnames of the resources. Resources whose shortnames are included in this array will be enabled by default. In case this array is left emply, all resources will be enabled by default.

After creating catalogue file and placing it in /data/catalogues directory we will see no change in Library app. But if we visit Collections page, the name of the new collection will be visible in the table.

Listing collection resources

The catalogue just created is empty because no resource is listed in it. Library app supports following types of resources: Parsed text, Critical edition, Concordance, Lexicon, Translation, Commentary, Grammar references, Diagram set, Audio recording. In the catalogue code above you see names of the resource types as keys of JSON object.

Parsed text

There can be only one parsed text inside collection and parsed text resource is always transcribed, never scanned.

As an example let‘s list parsed text resource for Epistle to Titus collection:

, "parsed_text": {
  
"resource_shortname": "nt_titus_parsed_text" 
      
, "collections_page_resource_desc": "compiled by J. K. Tauber (2016)" 
      
, "library_app_panel_title": "Ἡ πρὸς Τίτον ἐπιστολή"
      
, "library_app_panel_unit_num": ""
      
, "library_app_panel_text_from": "The New Testament in the original Greek. Byzantine textform. Compiled and arranged by M. A. Robinson and W. G. Pierpont. 2005. Southborough, MA: Chilton book publishing."
      
, "library_app_panel_note": "Parsing by J. K. Tauber (2016)."
      
, "scanned_or_typed": "typed"
      
, "scanned_source_shortname": "new_testament_in_original_greek_ed_robinson_pierpont" // if not empty, must be different from resource_shortname
      
 , "contents_shortname": ""
  
}

After you add these lines into collection catalogue‘s file, you should see parsed text title in Parsed text pane and info about parsed text in the table in Collections page.

JSON object‘s key "resource_shortname" should have as its value a string which has at the beginning collection shortname and the string "_parsed_text" at the end.

The key "collections_page_resource_desc" should have as its value a string "compiled by" plus compilator‘s surname and the year in brackets.

The key "library_app_panel_title" should have as its value the title of the collection in original language.

The key "library_app_panel_unit_num" should have as its value name and number in original language of the part of the book. In the example this key has as its value empty string because the text comprises whole text of the Epistle to Titus. But other collection may inlude only part of the text of some classical work, usualy a single book. So, for example there is collection Xenophon. Anabasis. Book 1, which includes only one book. The value of the key "library_app_panel_unit_num" is in this case the string "Βιβλίον Αʹ".

The key "library_app_panel_text_from" should have as its value bibliographich description of the book from which the text for parsing was taken.

The key "library_app_panel_note" should have as its value the string "Parsing by" plus the surname plus the year in brackets.

The key "scanned_or_typed" should have as its value the string "typed" (it means "transcribed").

The key "scanned_source_shortname" should have as its value the name of the directory (without path) in which has been uploaded the images of the pages of the book from which the text for parsing was taken. Directory name includes title of the book, string "_ed_" and surnames of the authors. The value for this key must be different from the value for the key "resource_shortname".

The key "contents_shortname" should have as its value empty string because parsed text does not have its own content table; it uses collection‘s content table. Collection content‘s shortname was defined when empty catalogue was created.

There can be many critical editions inside collection. In principle, resources belonging to type "Critical edition" can be either scanned or transcribed, but they surely will be all scanned because it is very difficult to reproduce complex formatting of the scholarly editions using very limitted means available in HTML.

Creating resources

Library app supports following types of resources: Parsed text, Critical edition, Concordance, Lexicon, Translation, Commentary, Diagram set, Audio recording. Parsed text, Diagram set and Audio recording resources can only be transcribed; Critical edition, Concordance, Lexicon, Translation, Commentary resources can be transcribed or scanned. Creating transcribed and scanned resources will be described separately because the process is very different.

Parsed text (transcribed)

Creating Parsed text resource involves creating two files: contents file and data file. The content file of the Parsed text resource also serves as the content file for the whole collection.

Creating contents file

The contents file is being created by filling Collection contents generator form.

Let‘s create contents file for Epistle to Titus collection.

The value of the form‘s field "Contents shortname" should be the same as the value of the key "contents_shortname" in collection‘s catalogue. While creating collection‘s catalogue for Epistle to Titus collection we entered for this key the value "coll_nt_titus_contents", so we now should enter it as the value of the field "Contents shortname".

The value of the form‘s field "Line range" should be pipe delimited line ranges. In Epistle to Titus there are three chapters, the first having 16 verses, the second – 15 verses and the third – 15 verses. So, the value of this field should be 1-16|1-15|1-15.

The value of the form‘s field "Line number prefix" should be pipe delimited line prefixes, one for each chapter. Line number prefix usually indicates the number of the chapter. The chapters of the New Testament are usually indicated by the number; colon usually used to separate chapter number from verse number. So, the value of this field should be 1:|2:|3: .

The value of the form‘s field "Line class prefix" should be be pipe delimited class prefixes, one for each chapter. Class prefix should start with the letter "v" and follow with line number prefix in which special symbols are replaced with dashes. So, we replace colons in line number prefixes, add leading letter "v" and get the value for this field v1-|v2-|v3- .

Tags and required attributes

Data file uses certain formatting conventions.

HTML "span" tag having class attribute "verse" should wrap each line of the text.

"span" tag having class attribute "verse-number" should wrap verse number in each line. The same "span" element should also have as its class attribute the line class as generated by contents generator.

"span" tag having class attribute "word" should wrap each word. The same "span" element should also have the attributes "data-form", "data-part-of-speech", "data-parsing", "data-lemma". These four attributes contain morphological data of the word.

"span" tag having class attribute "para" should mark start of each new paragraph.

HTML code formatted using these "span" tags should be converted to string and made the second argument of the function "ParsedClassicsData._loadResource". The first argument of this function should be the value of the key "resource_shortname" from collection catalogue. Parsed text resource shortname from Epistle to Titus catalogue is "nt_titus_parsed_text", so this string will be the first argument.

Optional attributes

It happens, that more than one word has the same lemma. E.g. in Greek-English lexicon to the New Testament by Th. S. Green two words have the lemma δέω, two words have the lemma μήν, and two words have the lemma ὅτι. If one clicks in parsed text pane on the word whose lemma has more than one article in the dictionary, then by default the first article will be crolled into view.

In cases second, third etc. lexicon article should be loaded we need to use two additional attributes for the tag "span" having class "word". These are attributes "data-lexicon" and "data-lexicon-entry". The value of the attribute "data-lexicon" should be resource shortname of the relevant lexicon; the value of the attribute "data-lexicon-entry" should be order number of the entry which should be scrolled into view. E.g. the verse Matthew 9:38 contains the word δεήθητε, whose lemma is δέω. The needed article in Green's lexicon is second article for the lemma δέω. To load second article the code for the word δεήθητε in parsed text resource should be as this:

' <span class="word" data-form="δεήθητε" data-part-of-speech="verb" data-parsing="aorist passive imperative 2nd person plural" data-lemma="δέω" data-lexicon="greek_lexicon_to_nt_by_green" data-lexicon-entry="2">δεήθητε</span>' +

Parsing scheme

"span" tag having class attribute "word" should wrap each word in the data file. This "span" tag also has the attributes "data-form", "data-part-of-speech", "data-parsing", "data-lemma which contain morphological data of the word.

The attribute "data-form" contains as its value the same form of the word which "span" tag wraps. Usually the word wrapped and the value of "data-form" are exactly the same. But in some cases, as word undergoing elision or word having movable consonant or word having several orthographic variants the word wrapped and the value of "data-form" will be slightly different.

The attribute "data-lemma" contains as its value the dictionary lemma of the wrapped word.

The attribute "data-part-of-speech" contains as its value the name of the part of speech the wrapped word belongs to.

The attribute "data-parsing" contains as its value parsing data of the wrapped word except the part of speech.

The values of the attributes "data-part-of-speech" and "data-parsing" are entered according to parsing scheme. The attribute "data-part-of-speech" always must have a value and always contains one piece of data. The attribute "data-parsing" can be empty or contain one or more pieces of data. In case the attribute "data-parsing" contains several pieces of data, these pieces should be in proper order, i.e. each piece has its own position. E.g. the value of the attribute "data-parsing" for nouns should consist of case, number and gender exactly in this order.

Coding and parsing policy

Dealing with missing and differently numbered lines

It happens, that some lines of the classical text, which are included in some critical editions, are considered inauthentic and excluded from other editions. E.g. lines 16:25-27 of apostle Paul's epistle to Romans are included in Scrivener's edition, in Westcott and Hort's edition, and in Nestle and Nestle's edition of the New Testament, but are excluded from Robinson and Pierpont's edition and from Antoniadis's edition; in Westcott and Hort's edition and in Nestle and Nestle's edition of the New Testament the line 16:24 is not included.

So, it may be that the critical edition parsed text resource is based on lacks certain lines included in some other critical edition. In such case parsed text resource should include the line numbers of missing lines. In the parsed text of the apostle Paul's epistle to Romans which is based on Robinson and Pierpont's edition of the New Testament, the lines 16:25-27 are represented in parsed text resource's data file by their line numbers:

' <span class="verse"><span class="verse-number v16-25">16:25</span></span>' +


' <span class="verse"><span class="verse-number v16-26">16:26</span></span>' +


' <span class="verse"><span class="verse-number v16-27">16:27</span></span>' +

After including numbers of missing lines in parsed text resource's data file we should not forget to to add these same lines into contents file.

Also it happens that some lines of the classical text are numbered slightly differently in some critical editions. E.g. the line from apostle John's Third Catholic epistle, which in Robinson and Pierpont's edition of the New Testament is numbered as 1:14, in Antoniadis's edition, in Scrivener's edition, in Westcott and Hort's edition and in Nestle and Nestle's edition is numbered as lines 1:14-15. In cases when numbering of lines has some differences across critical editions, it is best for parsed text to accept the most common numbering of lines.

Dealing with crasis

Cases of crasis, such as κἀκεῖ, κἀκεῖθεν, κἀκεῖνος, κἄν, ταὐτά, τοὐναντίον, τοὔνομα, τουτέστι, although might be explained in certain dictionaries, should be treated as two words and those two words should be parsed separately. The "span" tag of the first word should contain that part of the crasis expression which is before vowel having breathing mark, and the "span" tag of the second word should contain part of the crasis expression strarting with vowel having breathin mark to the end of expression. E.g. in case of crasis expression κἀκεῖθεν the "span" tag of the first word will contain "κ" and the "span" tag of the second word will contain "ἀκεῖθεν". The code for crasis expression κἀκεῖθεν will be this:

' <span class="word" data-form="καί" data-part-of-speech="conjunction" data-parsing="" data-lemma="καί">κ</span><span class="word" data-form="ἐκεῖθεν" data-part-of-speech="adverb" data-parsing="" data-lemma="ἐκεῖθεν">ἀκεῖθεν</span>' +

Dealing with alternative characters

In unicode two or more code points may encode characters which look identically or have only stylistic difference. In order for all functionality to work correctly the uniformity of encoding must be enforced by using only characters from certain "standard" set. To achieve this go to Unicode Characters changer and use the form to change unicode characters into "standard" ones.

Critical edition (scanned)

In principle, resources belonging to type "Critical edition" can be either scanned or transcribed, but they surely will be all scanned because it is very difficult to reproduce complex formatting of the scholarly editions using very limitted means available in HTML. So, only creation of scanned resources of the type "Critical edition" will be described in documentation.

Creating image files

To create scanned resource of the type "Critical edition" you first need a series of JPG images, each image picturing one page of the book. Scanned resource always includes all pages of the book, i.e. each page from the title page to the last, not including covers and promotional materials. About creating a series of JPG files for scanned resource, read in Common tasks the sections on scanning a book, on converting PDF file to images, on batch renaming of files, on cropping images, on retouching images, on optimizing size of JPG files.

When you have a series of JPG files each image picturing one page of the book and named sttarting from page0001.jpg, submit them to the senior editor who will upload those files to the server. The files will be uploaded to the directory specified in catalogue file as the value of the key "scanned_source_shortname". For example, in the Epistle to Titus collection the critical edition by M. W. Holmes has been listed and the value of the key "scanned_source_shortname" is "greek_new_testament_sbl_edition_ed_holmes". Workflows for multiple collections

Usually scanned resource of the type "Concordance" is being included into several collections. E.g. such resource as concordance edited by W. F. Moulton and A. S. Geden will be used in every collection dedicated to certain book of the New Testament. This means that the same contents file of the scanned resource of the type "Concordance" will be used by all relevant collections.

There are several workflows for creating contents file to be used by several collections.

Supplementing in stages

One possible workflow is as follows: at first we create new contents file to be used by certain single collection and fill the gaps with page numbers for all lemmas found in the original text of that collection. Then we supplement the contents file with the lemmas found in original text of another collection and fill the gaps with page numbers of these lemmas. And we repeat this process again and again untill all lemmas of the all original texts of the all relevant collections are included in the contents file and all relevant page numbers are filled in.

This workflow has the advantage that we can work only with lemmas of the original text of the single collection and doing so quickly prepare one or more resources of the type "Concordance" to be used inside single collection. The disadvantage is that in each cycle we need again to browse through all pages of the scanned book in search for relevant lemmas and that takes a lot of time.

Supplementing at start

Another possible workflow is as follows: at first we create new contents file to be used by certain single collection. Then we don't fill the gaps with page numbers, but instead supplement the contents file with the lemmas found in original text of another collection. And we repeat this process again and again untill all lemmas of the all original texts of the all relevant collections are included in the contents file. Then we browse through all pages of the scanned book, find the page numbers of the all lemmas and fill the gaps in contents file with these page numbers.

This workflow has the advantage that we must browse though all pages of the scanned book only once and this takes much less time than repeated browsing of the workflow described above. But the disadvantage is that we cannot work with the single collection at the time and must create huge contents file at once.

Cleaning page numbers

By using the first or the second workflow of those described above we get contents file of the resource of the type "Concordance" which is being used by multiple collections. We may need to add another resource of the type "Concordance" to the same collections. E.g. after having added concordance edited by W. F. Moulton and A. S. Geden to every collection dedicated to certain book of the New Testament we may need to add concordance edited by Ch. F. Hudson and E. Abbot to the same collections. In such cases we can shortcut the process by using the third workflow.

The contents file for the new resource of the type "Concordance" should contain the same lemmas. This means that we can reuse existing contens file: we only need to remove page numbers, to change the name of the contents object and to change the name of the file. To remove page numbers from the contents file use Lexicon page numbers cleaner. Copy contents object from the contens file, paste it in the field "Lexicon contents object" and click the button "Clean page numbers":

Then save the outputted contens object without page numbers using relevant contents object name and file name.

Let's say we want to compile grammatical concordance for all books of the New Testament starting from Gospel according to Matthew and ending with Revelation to John the Theologian. The value of the form‘s field "Concordance to be supplemented" should be the code of the concordance of the Gospel according to Mark.

Workflows for multiple collections

Usually scanned resource of the type "Lexicon" is being included into several collections. E.g. such resource as E. Robinson's Greek and English lexicon of the New Testament will be used in every collection dedicated to certain book of the New Testament. This means that the same contents file of the scanned resource of the type "Lexicon" will be used by all relevant collections.

There are several workflows for creating contents file to be used by several collections.

Supplementing in stages

One possible workflow is as follows: at first we create new contents file to be used by certain single collection and fill the gaps with page numbers for all lemmas found in the original text of that collection. Then we supplement the contents file with the lemmas found in original text of another collection and fill the gaps with page numbers of these lemmas. And we repeat this process again and again untill all lemmas of the all original texts of the all relevant collections are included in the contents file and all relevant page numbers are filled in.

This workflow has the advantage that we can work only with lemmas of the original text of the single collection and doing so quickly prepare one or more resources of the type "Lexicon" to be used inside single collection. The disadvantage is that in each cycle we need again to browse through all pages of the scanned book in search for relevant lemmas and that takes a lot of time.

Supplementing at start

Another possible workflow is as follows: at first we create new contents file to be used by certain single collection. Then we don't fill the gaps with page numbers, but instead supplement the contents file with the lemmas found in original text of another collection. And we repeat this process again and again untill all lemmas of the all original texts of the all relevant collections are included in the contents file. Then we browse through all scanned pages of the scanned book, find the page numbers of the all lemmas and fill the gaps in contents file with these page numbers.

This workflow has the advantage that we must browse though all pages of the scanned book only once and this takes much less time than repeated browsing of the workflow described above. But the disadvantage is that we cannot work with the single collection at the time and must create huge contents file at once.

Cleaning page numbers

By using the first or the second workflow of those described above we get contents file of the resource of the type "Lexicon" which is being used by multiple collections. We may need to add another resource of the type "Lexicon" to the same collections. E.g. after having added E. Robinson's Greek and English lexicon of the New Testament to every collection dedicated to certain book of the New Testament we may need to add F. Zorell's Lexicon Graecum Novi Testamenti to the same collections. In such cases we can shortcut the process by using the third workflow.

The contents file for the new resource of the type "Lexicon" should contain the same lemmas. This means that we can reuse existing contens file: we only need to remove page numbers, to change the name of the contents object and to change the name of the file. To remove page numbers from the contents file use Lexicon page numbers cleaner. Copy contents object from the contens file, paste it in the field "Lexicon contents object" and click the button "Clean page numbers":

Then save the outputted contens object without page numbers using relevant contents object name and file name.

At first we should take that bigger contents file which contains comprehensive list of lemmas found in multiple collections and remove the page numbers from it. The following conventions are used to format lexicon articles.

(1) On the top of the series of articles on words starting from certain letter is that letter wrapped by "h3" tag.

(2) Series of articles on words starting from certain letter is wrapped by "dl" tag.

(3) The words which the lexicon explains are wrapped by "dt" tag.

(4) The lemmas of explained words are wrapped by "span" tags having "class" attribute value "lexicon-heading". These "span" tags have "data-lemma" attribute. The value of this attribute is lowercase lemma of the explained word.

(5) Explanation of the word is wrapped by "dd" tag. Certain word may have several explanations, so below "dt" tag may stand several "dd" tags.

(6) Inner links, i.e. the links that lead from one place in the lexicon to another place in the same lexicon are wrapped by "a" tags. Inner link anchor has "class" attribute value "pc-anchor" and "id" attribute value consisting from lowercase lemma prefixed by certain string indicating lexicon title. Inner link himself has "class" attribute value "inner-link" and "data-anchor" attribute value consisting from lowercase lemma prefixed by certain string indicating lexicon title. In code example above there is link <a class="inner-link" data-anchor="green-nt-lex-βαρος" >βάρος</a> which leads to its anchor <a class="pc-anchor" id="green-nt-lex-βαρος" >Βάρος</a>.

Using these conventions lexicon articles should be formatted. There are no definite conventions how to format introduction, table of contents, various indexes, etc. Ask about it senior editor.

Coding policy

Dealing with alternative characters

In unicode two or more code points may encode characters which look identically or have only stylistic difference. In order for all functionality to work correctly the uniformity of encoding must be enforced by using only characters from certain "standard" set. To achieve this go to Unicode Characters changer and use the form to change unicode characters into "standard" ones.

Translation (scanned)

Creating image files

To create scanned resource of the type "translation" you first need a series of JPG images, each image picturing one page of the book. Scanned resource always includes all pages of the book, i.e. each page from the title page to the last, not including covers and promotional materials. About creating a series of JPG files for scanned resource, read in Common tasks the sections on scanning a book, on converting PDF file to images, on batch renaming of files, on cropping images, on retouching images, on optimizing size of JPG files.

When you have a series of JPG files each image picturing one page of the book and named starting from page0001.jpg, submit them to the senior editor who will upload those files to the server. The files will be uploaded to the directory specified in catalogue file as the value of the key "scanned_source_shortname". For example in the Epistle to Titus collection the translation to Latin by Saint Jerome has been listed and the value of the key "scanned_source_shortname" is "novum_testamentum_latine_ed_brandscheid". It means that the pictures of the pages must be uploaded to the directory /_scans/novum_testamentum_latine_ed_brandscheid .

Having uploaded the images of the pages to the server, senior editor should list the scanned book as described in section Listing scanned books.

After uploading the files of the scanned book and listing it among scanned books, it can be browsed using the value of the key "scanned_source_shortname" in the URL: parsedclassics.com/reader/embedded_bookreader.html?novum_testamentum_latine_ed_brandscheid

Creating syntax diagrams

Syntax diagrams are being created in OmniGraffle program by dragging objects from ParsedClassics stencils to the canvas, joining them and filling text boxes.

Notice, that in stencils some objects look identical and are put into pairs. Identical looking objects are not the same: the upper one has straight line and the one below has curved line. Accordingly, use the upper when you need to draw straight line and the one below – when you need to draw curved line.

Text boxes in the stencil objects are indicated by

(1) uppercase letter ending with dot; in this case the word should be pasted on the right from the dot;
(2) uppercase letter "W"; in this case the letter "W" should be substituted with word;
(3) lowercase letter; lowercase letter is placeholder for index and should be replaced with other lowercase letter.

The diagram created on OmniGraffle canvas should be saved both as OmniGraffle document and JPG file.

The saving of created diagram is done using "artboard". Click in the toolbar the button "Artboard Tool" and select the area in the canvas which should be visible in diagram image:

Then go File > Save As and save the diagram as OmniGraffle document. The file name should contain short title of the text, chapter and line numbers. For numbers zero padding should be used: to 2 for chapters and to 3 for line numbers. For example, syntax diagram which displays lines 1-5 of the first chapter of the Epistle to Titus should be named "nt_titus_01_001-005".

After that export the diagram as JPG image. Go File > Export and in the opened modal dialogue window select the tab "JPG" and click the button "Export".

Scanning a book

The book needed for scanned resource should be scanned in full (pages from the title page to the last, not including covers and promotional materials). Each page should be scanned into separate JPG file. Scanning resolution should be at least 400dpi, scanning mode should be "color" or "greyscale".

Quite often it is more convenient to scan two pages of the book into one JPG file. In this case you will need to split each file into two files. This can be done usinge "TileMage" program (Win, free). This program places each pair of new files created into separate directory. So you will need to flatten directory structure. This can be made using program "Suction" (Win, free).

Using regex

While editing code of the file of the transcribed resource, there is often the need to remove or replace certain parts of the HTML code. For example, you may need to remove "id" or "href" attributes, to remove opening or closing part of certain tags, to reverse order of tags, to extract certain numbers and put them in another place. The most efficient way to do this is using regular expressions (regex).

Regular can be used with such code editors as Notepad++ (Win, free), ActiveState Komodo Edit (Win/Mac, free) and many others.

For using regular expressions with Notepad++ code editor go Search > Replace and inside the dialogue check "Regular expression", enter regex in the "Find" field, enter another expression in the field "Replace" or leave it blank if you need to delete and click "Replace All". Or click "Replace All in All Opened Documents" if you need to make replacements in several files. All strings which match the regular expression will be replaced:

For using regular expressions with ActiveState Komodo Edit code editor go Menu > Replace and inside the dialogue click the button "Treat the pattern as a regular expression", enter regex in the "Find" field, enter another expression in the field "Replace" or leave it blank if you need to delete and click "Replace all". All strings which match the regular expression will be deleted:

Find field Replace field Used for Tested with
