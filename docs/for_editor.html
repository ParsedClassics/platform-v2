Editor‘s documentation

This documentation explains the backend work needed to create ParsedClassics collections.

To understand adequately Editor‘s documentation you should first study the Reader‘s documentation.

Editorial roles

There are two editorial roles: senior editor and resource editor. Senior editor must know basics of HTML and Javascript coding.

Senior editor

Senior editor performs the following tasks:

- defines edition;
- defines collection and its resources;
- sends the files to work with to resource editors;
- explains tools to be used and formatting rules to resource editors;
- checks formatting of the files created by resource editors;
- uploads image files to the server;
- maintains Git repository;
- deploys code to the server.

Resource editor

Resource editor performs the following task:

- creates resource files.

Editor‘s main tools

All editing work is being done on the localhost. The main tools are coding editor to create and edit files and web browser to view the result.

In order to start to work on ParsedClassics project, the editor should place the copy of ParsedClassics app into some directory on localhost, open /classics.html file in web browser and fire the coding editor program. The editing work consists in making edits in the code editor, saving and previewing results in the web browser.

Coding editor software

Any HTML/CSS/Javascript coding editor software is suitable for ParsedClassics projects. We recommend VS Code (Win/Mac, free).

Greek polytonic keyboard

To type Greek characters Greek polytonic keyboard software should be used. It is recommended to use virtual Greek polytonic keyboard, because it implements encoding conventions required for resources of ParsedClassics collections.

Web browser

Any major modern webbrowser – Chrome, Firefox, Opera, Safari, Brave, MS Edge – is suitable for previewing ParsedClassics projects on localhost. Among them Firefox has certain advantage: it preserves form values while reloading the page. But in Firefox default security settings may prevent the webfonts to be rendered in localhost environment. To see correct font in Firefox in localhost environment go to about:config and change the value of security.fileuri.strict_origin_policy to "false":

Git client and code deployment tool

For maintaining and deploying the code senior will need Git client and deployment tool.

As simple and convenient Git client can be recommended Github desktop app (Win/Mac, free).

As simple and convenient code deployment tool the online service deployhq.com can be recommended.

Defining edition

To define a new edition senior editor should add JSON object into the file /data/_catalogues/_sets.js . The order of JSON objects in this file is important because Catalogue page displays the list of editions in the same order.

For example, to define the edition "Ἡ Καινὴ Διαθήκη. Πράξεις, Ἐπιστολαί, Ἀποκάλυψις / The New Testament. Acts, Epistles, Revelation" such JSON object has been added:

  nt_apostolics: {
    title_orig: 'Ἡ Καινὴ Διαθήκη. Πράξεις, Ἐπιστολαί, Ἀποκάλυψις',
    title_orig_short: 'Κ.Δ. Πράξεις, Ἐπιστολαί, Ἀποκάλυψις',
    title_eng: 'The New Testament. Acts, Epistles, Revelation',	
    collections: [],	
  },

The value of the key "title_orig" should be a string in original language. It should include author or scripture name and the title. Abbreviation or/and shortened title can be used in order to avoid very long value. This value will be used in Catalogue page.

The value of the key "title_orig_short" should be a string in original language. This value usually is shorter than the value of the key "title_orig". Abbreviation or/and shortened title can be used in order to avoid very long value. This value will be used inside Collections selectbox at Library app page.

The value of the key "title_eng" should be a string in English. It should include author or scripture name and the title. Abbreviation or/and shortened title can be used in order to avoid very long value. This value will be used in Catalogue page.

The value of the key "collections" should be array of strings which are shortnames of collections. For now let's leave this array empty; we will add member to this array after defining certain collection.

After editing /data/_catalogues/_sets.js file refresh Catalogue page. A new value should appear in editions list.

Defining collection and its resources

Defining collection and its resources consists of the following tasks:
(1) listing collection;
(2) registering collection as part of edition;
(3) creating empty collection catalogue;
(4) listing collection resources;
(5) listing scanned books.

Listing collection

To list a new collection senior editor should add JSON object into the file /data/_catalogues/_collections.js as the value of certain key of ParsedClassicsCollDefs object.

For example, to list the collection Epistle to Titus such JSON object has been added as the value of the key "nt_titus":

const ParsedClassicsCollDefs = {
  
  // some lines have been omitted

  nt_titus: {
    author_orig: 'Ἡ Καινὴ Διαθήκη',   
    author_eng: 'The New Testament',
    collection_selectboxname: 'Κ.Δ. Πρὸς Τίτον',
    collections_page_title_orig: 'Παύλου τοῦ ἀποστόλου ἡ πρὸς Τίτον ἐπιστολή',
    collections_page_title_eng: 'Apostle Paul\'s epistle to Titus', 
  },
  
  // some lines have been omitted

}

The key used to list collection let's call "collection shortname". This is important term and will be used often in this documentation.

Collection shortname should be a string containing only lowercase latin letters, numbers and underscores (no spaces!). It should include author or scripture name and the title. Abbreviation or/and shortened title can be used in order to avoid very long string.

The value of the key "collection_selectboxname" should be a string in original language. It should include author or scripture name and the title. Abbreviation or/and shortened title can be used in order to avoid very long value. It will be displayed in Collections selectbox.

The key "collections_page_title_eng" should have as its value the title of the collection in English.

The key "collections_page_title_orig" should have as its value the title of the collection in original language.

Both English and original title will be displayed in Collections page. Original title will also be displayed as title of parsed text resource inside Library app.

The key "author_orig" should have as its value the name of the author or the name of the scripture in original language, i. e. in Greek or Latin. It will be displayed as author name of parsed text resource inside Library app.

The key "author_eng" should have as its value English name of the author or English name of the scripture. The value is currently not used.

Since classics text usually consists of several books and each book usually will have its own collection, the values of the keys "author_orig" and "author_eng" will be repeated listing those collections. In order to avoid repetitions it is recommended to define JSON object containing all keys whose values would be repeated. That JSON object should be the value of certain key of ParsedClassicsCollProtos object which is in the same file as ParsedClassicsCollDefs object. Since the values of the keys "author_orig" and "author_eng" will be repeated for all NT books, let's name the key "nt_book":

  const ParsedClassicsCollProtos = {

    // some lines have been omitted

    nt_book: {
      author_orig: 'Ἡ Καινὴ Διαθήκη',   
      author_eng: 'The New Testament',
    },

    // some lines have been omitted
  
  };

Now we can destructure the object ParsedClassicsCollProtos.nt_book anywhere we need the same values of "author_orig" and "author_eng" keys:

  const ParsedClassicsCollDefs = {
    
    // some lines have been omitted
  
    nt_titus: {
      ...ParsedClassicsCollProtos.nt_book,
      collection_selectboxname: 'Κ.Δ. Πρὸς Τίτον',
      collections_page_title_orig: 'Παύλου τοῦ ἀποστόλου ἡ πρὸς Τίτον ἐπιστολή',
      collections_page_title_eng: 'Apostle Paul\'s epistle to Titus', 
    },
    
    // some lines have been omitted
  
  }

Registering collection

The collection we just listed among other collections should be registered as a part of edition. This should be done by entering collection shortname as member of array which is value of the key "collections" in edition's definition:

  nt_apostolics: {
    title_orig: 'Ἡ Καινὴ Διαθήκη. Πράξεις, Ἐπιστολαί, Ἀποκάλυψις',
    title_orig_short: 'Κ.Δ. Πράξεις, Ἐπιστολαί, Ἀποκάλυψις',
    title_eng: 'The New Testament. Acts, Epistles, Revelation',	
    collections: [
      'nt_titus',
    ],	
  },

The order of collection shortnames is important, because list of collections will be displayed in that order.

Creating empty collection catalogue

Collection catalogue is a file inside /data/_catalogues/ directory. Collection catalogue‘s filename should be the same as the collection shortname. In the example above collection shortname was "nt_titus". That means the name of the file of the collection catalogue should be "nt_titus.js".

The encoding of the collection catalogue file must be UTF-8 (no BOM).

Let‘s as example to create catalogue for Epistle to Titus collection.

To create empty catalogue for this collection we need to add to the object ParsedClassicsCollDefs the key "nt_titus" and to the object "nt_titus" the object "resource_defs"; let's also indicate the order of resource types:

ParsedClassicsCollDefs.nt_titus["resource_defs"] = {
  // Parsed text
  // IMPORTANT!
  // The contents of the first resource of the type "Parsed text" serves as contents of the whole collection
  
  // Original texts

  // Concordances

  // Lexicons

  // Translations

  // Commentaries

  // Commentary references

  // Grammar references

  // Diagram sets

  // Audio
}

Listing collection resources

The catalogue just created is empty because no resource is listed in it. Library app supports following types of resources: Parsed text, Critical edition, Concordance, Lexicon, Translation, Commentary, Commentary references, Grammar references, Diagram set, Audio recording.

Parsed text

As an example let‘s list parsed text resource for Epistle to Titus collection. Todo it we need to add JSON object as the value of certain key of ParsedClassicsCollDefs.nt_titus["resource_defs"] object. The key used to list resource let's call "resource shortname". This is important term and will be used often in this documentation.

In this case the key is "nt_titus_parsed_text" (contains inside collection shortname and resource type):

  ParsedClassicsCollDefs.nt_titus["resource_defs"] = {
    // Parsed text
    // IMPORTANT!
    // The contents of the first resource of the type "Parsed text" serves as contents of the whole collection

    nt_titus_parsed_text: {
      collections_page_resource_desc: "ed. by ParsedClassics (2018)",
      library_app_selectbox_title: "Byzantine textform", 
      library_app_panel_subtitle: "",
      library_app_panel_text_from: "The New Testament in the original Greek. Byzantine textform. Compiled and arranged by M. A. Robinson and W. G. Pierpont. 2005. Southborough, MA: Chilton book publishing.",
      library_app_panel_note: "",
      scanned_or_typed: "typed",
      resource_type: "parsed_text",
      scanned_source_shortname: "new_testament_in_original_greek_ed_robinson_pierpont",
      library_app_panel_title: "Παύλου τοῦ ἀποστόλου ἡ πρὸς Τίτον ἐπιστολή",
      contents_shortname: "nt_titus_parsed_text_contents",
    },
    
    // some lines have been omitted
  }

After you add these lines, you should see this resource listed in Resources selectbox and in Collections page.

The key "collections_page_resource_desc" should have as its value a string "ed. by" plus compilator‘s name and the year in brackets.

The key "library_app_panel_title" should have as its value the title of the resource in original language.

The key "library_app_panel_subtitle" should have as its value the subtitle of the resource in original language. Here this key has empty string value because resource has no subtitle.

The key "library_app_panel_text_from" should have as its value bibliographic description of the book from which the text for parsing was taken.

The key "library_app_panel_note" should have as its value some text. Most probably it will be some acknowledgments.

The key "scanned_or_typed" should have as its value the string "typed" (it means "transcribed").

The key "scanned_source_shortname" should have as its value the name of the directory (without path) in which have been uploaded the images of the pages of the book from which the text for parsing was taken. Directory name includes title of the book, string "ed" and names of the authors.

The key "library_app_selectbox_title" should have as its value short English title of the resource.

The key "resource_type" should have the value "parsed_text".

The key "contents_shortname" should have as its value the name of the file containing resource's contents. The name of this file is usually formed by adding the string "parsed_text" at the end of collection name and next adding the string "contents" or the string indicating resource and the string "contents". So, if the collection shortname was "nt_titus", the the name of parsed text resource contents file will be "nt_titus_parsed_text_contents" or "nt_titus_parsed_text_byzantine_contents". About creating contents file and placing it in directory will be explained below.

Since classics text usually consists of several books and each book usually will have its own collection with its own parsed text resource, the values of the many keys will be repeated listing those resources. In order to avoid repetitions it is recommended to define JSON object containing all keys whose values would be repeated. That JSON object should be the value of certain key of ParsedClassicsResProtos object which is in the file /data/_catalogues/_res_protos.js . Since the values of many keys will be repeated for all resources of the type "Parsed text" based on Byzantyne textform, let's name the key "nt_parsed_text_byzantine":

const ParsedClassicsResProtos = {
  // Parsed texts

  nt_parsed_text_byzantine: {
    collections_page_resource_desc: "ed. by ParsedClassics (2018)",
    library_app_selectbox_title: "Byzantine textform", 
    library_app_panel_subtitle: "",
    library_app_panel_text_from: "The New Testament in the original Greek. Byzantine textform. Compiled and arranged by M. A. Robinson and W. G. Pierpont. 2005. Southborough, MA: Chilton book publishing.",
    library_app_panel_note: "",
    scanned_or_typed: "typed",
    resource_type: "parsed_text",
    scanned_source_shortname: "new_testament_in_original_greek_ed_robinson_pierpont", 
  },

  // some lines have been omitted
}

Now we can destructure the object ParsedClassicsCollProtos.nt_parsed_text_byzantine anywhere we need the same values of its keys:

  ParsedClassicsCollDefs.nt_titus["resource_defs"] = {
    // Parsed text
    // IMPORTANT!
    // The contents of the first resource of the type "Parsed text" serves as contents of the whole collection

    nt_titus_parsed_text: {
      ...ParsedClassicsResProtos.nt_parsed_text_byzantine,
      library_app_panel_title: "Παύλου τοῦ ἀποστόλου ἡ πρὸς Τίτον ἐπιστολή",
      contents_shortname: "nt_titus_parsed_text_contents",
    },
    
    // some lines have been omitted
  }

Original text

There can be several original text resources inside collection. In principle, resources belonging to type "Original text" can be either scanned or transcribed, but they surely will be all scanned because it is very difficult to reproduce complex formatting of the scholarly editions using very limitted means available in HTML.

As an example let‘s list original text resource for Epistle to Titus collection. Todo it we need to add JSON object as the value of certain key of ParsedClassicsCollDefs.nt_titus["resource_defs"] object. In this case the key (resource shortname) is "nt_titus_text_ed_antoniadis" (contains inside collection shortname, resource type and editor's name):

ParsedClassicsCollDefs.nt_titus["resource_defs"] = {

    collections_page_resource_desc: "ed. by V. Antoniadis (1904)",

The key "library_app_panel_title" should have as its value short title of the resource in English.

The key "library_app_panel_subtitle" should have as its value short subtitle of the resource. Here this key has empty string value because resource has no subtitle.

The key "library_app_selectbox_title" should have as its value short English title of the resource.

The key "resource_type" should have the value "original_text".

Concordance

There can be many concordances inside a collection. Resource belonging to the type "Concordance" can be either scanned or transcribed.

As examples let‘s list scanned and transcribed concordance resources for Epistle to Titus collection. To do it we need to add JSON objects as the values of certain keys of ParsedClassicsCollDefs.nt_titus["resource_defs"] object. In this case the keys (resource shortnames) are "concordance_by_moulton_geden" and "nt_conc_by_parsedclassics" (contains inside resource type and editor's name):

ParsedClassicsCollDefs.nt_titus["resource_defs"] = {
  // some lines have been omitted

  // Concordances

  concordance_by_moulton_geden: {
    collections_page_resource_desc: "ed. by W. F. Moulton and A. S. Geden (1899)",

    scanned_or_typed: "typed",
    resource_type: "concordance",
    scanned_source_shortname: "",
    contents_shortname: "nt_conc_by_parsedclassics_contents",
  },

  // some lines have been omitted

}

After you add these lines, you should see these resources listed in Resources selectbox and in Collections page.

For scanned concordance resource the key "collections_page_resource_desc" should have as its value a string "ed. by" plus compilator‘s name and the year in brackets. For transcribed concordance resource the key "collections_page_resource_desc" should have as its value a string "compiled by" plus compilator‘s name and the year in brackets.

The key "library_app_panel_title" should have as its value short title of the resource in English.

The key "library_app_panel_subtitle" should have as its value short subtitle of the resource. Here this key has empty string value because resources has no subtitle.

The key "library_app_panel_text_from" should have as its value bibliographic description of the book.

The key "library_app_panel_note" should have as its value some text. Most probably it will be some acknowledgments. For transcribed concordance resource the value should be a string "Compiled by" plus compilator‘s name and the year in brackets.

The key "scanned_or_typed" should have as its value the string "scanned" for scanned resource and "typed" for transcribed resource.

The key "scanned_source_shortname" should have as its value the name of the directory (without path) in which have been uploaded the images of the pages of the book. Directory name includes title of the book, string "by" and names of the authors. For transcribed concordance resource the value of this key should be an empty string.

The key "library_app_selectbox_title" should have as its value short English title of the resource.

The key "resource_type" should have the value "concordance".

The key "contents_shortname" should have as its value the name of the file containing resource's contents. The name of this file is usually formed by adding the string "by" at the end of short title, then adding name of author or compiler and next adding the string "contents". About creating contents file and placing it in directory will be explained below.

Classics text usually consists of several books and each book usually will have its own collection. If we have certain concordance resource, it will be listed in all of these collections. That means there is no need to use ParsedClassicsResProtos object in listing concordance resources.

Listing scanned books

All scanned resources and most transcribed resources have non empty value for the key "scanned_source_shortname" when are listed as part of some collection. The value of this key is a name of the directory (without path) in which have been uploaded the images of the pages of the book.

The book should also be listed. The senior editor should add a key to object ParsedClassicsScannedBooks in the file /data/_catalogues/_scanned_books.js and a JSON object as the value of that key. The key to object ParsedClassicsScannedBooks should be no other than the value of "scanned_source_shortname" we entered while listing resource.

The JSON object‘s key "bookTitle" should have as its value scanned book title in book's language.

The key "bookUrlText" should have as its value all remaining bibliographic data of the book (author, edition, year, place, publishing house).

The key "thumbnail" should have as its value the filename of the image displaying the title page of the scanned book.

The key "label" should always have as its value the word "Title".

The key "value" should have as its value the string containing all relevant bibliographic data of the scanned book. Usually you will get it by combining values of the keys "bookTitle" and "bookUrlText".

Dealing with multi-volume books

Multi-volume scanned books should be listed as one book in case there are references from one volume to the other. Accordingly, all pages of multi-volume book should be uploaded into one directory and there should be only one entry in _scanned_books.js file.

Creating resources

Library app supports following types of resources: Parsed text, Original text, Concordance, Lexicon, Translation, Commentary, Commentary references, Grammar references, Diagram set, Audio recording. Parsed text, Commentary references, Grammar references, Diagram set, Audio recording resources can only be transcribed; Original text, Concordance, Lexicon, Translation, Commentary resources can be transcribed or scanned. Creating transcribed and scanned resources will be described separately because the process is very different.

Parsed text (transcribed)

Creating Parsed text resource involves creating two files: contents file and data file. The contents file of the first listed Parsed text resource also serves as the contents file for the whole collection.

It happens, that more than one word has the same lemma. E.g. in Greek-English lexicon to the New Testament by Th. S. Green two words have the lemma δέω, two words have the lemma μήν, and two words have the lemma ὅτι. If one clicks in parsed text resource on the word whose lemma has more than one article in the dictionary, then by default the first article will be crolled into view.

In cases second, third etc. lexicon article should be loaded we need to use two additional attributes for the tag "span" having class "word". The value of the attribute "data-lexicon" should be resource shortname of the relevant lexicon; the value of the attribute "data-lexicon-entry" should be order number of the entry which should be scrolled into view. E.g. the verse Matthew 9:38 contains the word δεήθητε, whose lemma is δέω. The needed article in Green's lexicon is second article for the lemma δέω. ` <span class="word" data-form="δεήθητε" data-part-of-speech="verb" data-parsing="aorist passive imperative 2nd person plural" data-lemma="δέω" data-lexicon="greek_lexicon_to_nt_by_green" data-lexicon-entry="2">δεήθητε</span>`

Parsing scheme

"span" tag having class attribute "word" should wrap each word in the data file. This "span" tag also has the attributes "data-form", "data-part-of-speech", "data-parsing", "data-lemma which contain morphological data of the word.

The attribute "data-form" contains as its value the same form of the word which "span" tag wraps. Usually the word wrapped and the value of "data-form" are exactly the same. But in some cases, as word undergoing elision or word having movable consonant or word having several orthographic variants, the word wrapped and the value of "data-form" will be slightly different.

The attribute "data-lemma" contains as its value the dictionary lemma of the wrapped word.

The attribute "data-part-of-speech" contains as its value the name of the part of speech the wrapped word belongs to.

The attribute "data-parsing" contains as its value parsing data of the wrapped word except the part of speech.

The values of the attributes "data-part-of-speech" and "data-parsing" are entered according to parsing scheme. The attribute "data-part-of-speech" always must have a value and always contains one piece of data. The attribute "data-parsing" can be empty or contain one or more pieces of data. In case the attribute "data-parsing" contains several pieces of data, these pieces should be in proper order, i.e. each piece has its own position. E.g. the value of the attribute "data-parsing" for nouns should consist of case, number and gender exactly in this order.

Coding and parsing policy

Dealing with missing and differently numbered lines

It happens, that some lines of the classical text, which are included in some critical editions, are considered inauthentic and excluded from other editions. E.g. lines 16:25-27 of apostle Paul's epistle to Romans are included in Scrivener's edition, in Westcott and Hort's edition, and in Nestle and Nestle's edition of the New Testament, but are excluded from Robinson and Pierpont's edition and from Antoniadis's edition; in Westcott and Hort's edition and in Nestle and Nestle's edition of the New Testament the line 16:24 is not included.

So, it may be that the critical edition parsed text resource is based on lacks certain lines included in some other critical edition. In such case parsed text resource should include the line numbers of missing lines. In the parsed text of the apostle Paul's epistle to Romans which is based on Robinson and Pierpont's edition of the New Testament, the lines 16:25-27 are represented in parsed text resource's data file by their line numbers:

`<span class="line"><span data-line-number="16:25" class="line-number">16:25</span></span>


<span class="line"><span data-line-number="16:26" class="line-number">16:26</span></span>


<span class="line"><span data-line-number="16:27" class="line-number">16:27</span></span>`

After including numbers of missing lines in parsed text resource's data file we should not forget to to add these same lines into contents file.

Also it happens that some lines of the classical text are numbered slightly differently in some critical editions. E.g. the line from apostle John's Third Catholic epistle, which in Robinson and Pierpont's edition of the New Testament is numbered as 1:14, in Antoniadis's edition, in Scrivener's edition, in Westcott and Hort's edition and in Nestle and Nestle's edition is numbered as lines 1:14-15. In cases when numbering of lines has some differences across critical editions, it is best for parsed text to accept the most common numbering of lines.

Dealing with crasis

Cases of crasis, such as κἀκεῖ, κἀκεῖθεν, κἀκεῖνος, κἄν, ταὐτά, τοὐναντίον, τοὔνομα, τουτέστι, although might be explained in certain dictionaries, should be treated as two words and those two words should be parsed separately. The "span" tag of the first word should contain that part of the crasis expression which is before vowel having breathing mark, and the "span" tag of the second word should contain part of the crasis expression starting with vowel having breathing mark to the end of expression. E.g. in case of crasis expression κἀκεῖθεν the "span" tag of the first word will contain "κ" and the "span" tag of the second word will contain "ἀκεῖθεν". The code for crasis expression κἀκεῖθεν will be this:

`<span class="word" data-form="καί" data-part-of-speech="conjunction" data-parsing="" data-lemma="καί">κ</span><span class="word" data-form="ἐκεῖθεν" data-part-of-speech="adverb" data-parsing="" data-lemma="ἐκεῖθεν">ἀκεῖθεν</span>`

Dealing with alternative characters

In unicode two or more code points may encode characters which look identically or have only stylistic difference. In order for all functionality to work correctly the uniformity of encoding must be enforced by using only characters from certain "standard" set. To achieve this go to Unicode Characters changer and use the form to change unicode characters into "standard" ones.

Original text (scanned)

In principle, resources belonging to type "Original text" can be either scanned or transcribed, but they surely will be all scanned because it is very difficult to reproduce complex formatting of the scholarly editions using very limitted means available in HTML. So, only creation of scanned resources of the type "Critical edition" will be described in Editor's documentation.

Creating image files

To create scanned resource of the type "Critical edition" you first need a series of JPG images, each image picturing one page of the book. Scanned resource always includes all pages of the book, i.e. each page from the title page to the last, not including covers and promotional materials. About creating a series of JPG files for scanned resource, read in Common tasks.

When you have a series of JPG files each image picturing one page of the book and named starting from page0001.jpg, submit them to the senior editor who will upload those files to the server. The files will be uploaded to the directory which was entered as value of the key "scanned_source_shortname" when original text resource was listed as part of collection. For example, in the Epistle to Titus collection the original text edited by V. Antoniadis has been listed and the value of the key "scanned_source_shortname" is "kaine_diatheke_ed_antoniadis". It means that the pictures of the pages must be uploaded to the directory /_scans/kaine_diatheke_ed_antoniadis/ .

Having uploaded the images of the pages to the server, senior editor should list the scanned book as described in section Listing scanned books.

After uploading the files of the scanned book and listing it among scanned books, it can be browsed using the value of the key "scanned_source_shortname" in the URL: parsedclassics.com/reader/embedded_bookreader.html?kaine_diatheke_ed_antoniadis .

Workflows for multiple collections

Usually scanned resource of the type "Concordance" is being included into several collections. E.g. such resource as concordance edited by W. F. Moulton and A. S. Geden will be used in every collection dedicated to certain book of the New Testament. This means that the same contents file of the scanned resource of the type "Concordance" will be used by all relevant collections.

There are several workflows for creating contents file to be used by several collections.

Supplementing in stages

One possible workflow is as follows: at first we create new contents file to be used by certain single collection and fill the gaps with page numbers for all lemmas found in the parsed text of that collection. Then we supplement the contents file with the lemmas found in parsed text of another collection and fill the gaps with page numbers of these lemmas. And we repeat this process again and again until all lemmas of the all parsed texts of the all relevant collections are included in the contents file and all relevant page numbers are filled in.

This workflow has the advantage that we can work only with lemmas of the parsed text of the single collection and doing so quickly prepare one or more resources of the type "Concordance" to be used inside single collection. The disadvantage is that in each cycle we need again to browse through all pages of the scanned book in search for relevant lemmas and that takes a lot of time.

Supplementing at start

Another possible workflow is as follows: at first we create new contents file to be used by certain single collection. Then we don't fill the gaps with page numbers, but instead supplement the contents file with the lemmas found in original text of another collection. And we repeat this process again and again until all lemmas of the all original texts of the all relevant collections are included in the contents file. Then we browse through all pages of the scanned book, find the page numbers of the all lemmas and fill the gaps in contents file with these page numbers.

This workflow has the advantage that we must browse though all pages of the scanned book only once and this takes much less time than repeated browsing of the workflow described above. But the disadvantage is that we cannot work with the single collection at the time and must create huge contents file at once.

Cleaning page numbers

By using the first or the second workflow of those described above we get contents file of the resource of the type "Concordance" which is being used by multiple collections. We may need to add another resource of the type "Concordance" to the same collections. E.g. after having added concordance edited by W. F. Moulton and A. S. Geden to every collection dedicated to certain book of the New Testament we may need to add concordance edited by Ch. F. Hudson and E. Abbot to the same collections. In such cases we can shortcut the process by using the third workflow.

The contents file for the new resource of the type "Concordance" should contain the same lemmas. This means that we can reuse existing contents file: we only need to remove page numbers, to change the name of the contents object and to change the name of the file. To remove page numbers from the contents file use Lexicon page numbers cleaner. Copy contents object from the contents file, paste it in the field "Lexicon contents object" and click the button "Clean page numbers":

Then save the outputted contents object without page numbers using relevant variable name and file name.

Importing after cleaning

It may happen that one resource of the type "Concordance" has been added to multiple collections and another resource of the type "Concordance" has been added to just single or several collections. E.g. such resource as concordance edited by W. F. Moulton and A. S. Geden has been added to every collection dedicated to certain book of the New Testament, but such resource as concordance edited by Ch. F. Hudson and E. Abbot has been added to one or couple of collections. Now we need to add this resource to remaining collections.

In this case we need to use the following workflow. At first we should take that bigger contents file which contains comprehensive list of lemmas found in multiple collections and remove the page numbers from it. After that we need to import data into cleaned contents object from the contents object which contains less comprehensive list of lemmas from single collection or couple of collections. To do this go to Lexicon contents importer form and fill the fields:

The value of the form‘s field "Language" should be "Greek" because lemmas in both contents objects are in Greek.

The value of the field "Lexicon contents object" should be contents object in which page numbers have been removed, i.e. that contents object which contains comprehensive list of lemmas found in multiple collections.

The value of the field "Lexicon contents object to be imported" should be contents object with all page numbers intact which contains less comprehensive list of lemmas from single collection or couple of collections.

After filling fields with such values click the button "Load contents" which is below the first textarea and the list of lemmas contained in the initial contents object will be displayed below the heading "Initial contents object". Then click the button "Load contents" which is below the second textarea and the list of lemmas contained in contents object to be imported will be displayed below the heading "Contents object to be imported". After that click the button "Import contents" and the resulting contents object will be outputted inside textarea "Resulting contents object code". Then click the button "Check for completeness". All lemmas below the headings "Contents object to be imported" and "Initial contents object" should be colored green. If certain lemmas remain not colored, inform about it senior editor.

Then save the resulting contents object using relevant object name and file name.

Lexicon (scanned)

Creating image files

To create scanned resource of the type "Lexicon" you first need a series of JPG images, each image picturing one page of the book. Scanned resource always includes all pages of the book, i.e. each page from the title page to the last, not including covers and promotional materials. About creating a series of JPG files for scanned resource, read in Common tasks.

When you have a series of JPG files each image picturing one page of the book and named starting from page0001.jpg, submit them to the senior editor who will upload those files to the server. The files will be uploaded to the directory which was entered as value of the key "scanned_source_shortname" when lexicon resource was listed as part of collection. For example, in the Epistle to Titus collection the lexicon by E. Robinson has been listed and the value of the key "scanned_source_shortname" is "greek_and_english_lexicon_of_nt_by_robinson". It means that the pictures of the pages must be uploaded to the directory /_scans/greek_and_english_lexicon_of_nt_by_robinson/ .

Having uploaded the images of the pages to the server, senior editor should list the scanned book as described in section Listing scanned books.

After uploading the files of the scanned book and listing it among scanned books, it can be browsed using the value of the key "scanned_source_shortname" in the URL: parsedclassics.com/reader/embedded_bookreader.html?greek_and_english_lexicon_of_nt_by_robinson .

In order for lexicon to work in sync with parsed text, the contents file should be prepared. The same lexicon resource can be listed as part of several collections. This means that there could be two cases: (1) there is no contents file and we need to create it; (2) there is contents file, but the words of certain classics book are not fully dealt with, so we need to supplement it.

Workflows for multiple collections

Usually scanned resource of the type "Lexicon" is being included into several collections. E.g. such resource as lexicon by E. Robertson will be used in every collection dedicated to certain book of the New Testament. This means that the same contents file of the scanned resource of the type "Lexicon" will be used by all relevant collections.

There are several workflows for creating contents file to be used by several collections.

Supplementing in stages

One possible workflow is as follows: at first we create new contents file to be used by certain single collection and fill the gaps with page numbers for all lemmas found in the parsed text of that collection. Then we supplement the contents file with the lemmas found in parsed text of another collection and fill the gaps with page numbers of these lemmas. And we repeat this process again and again until all lemmas of the all parsed texts of the all relevant collections are included in the contents file and all relevant page numbers are filled in.

This workflow has the advantage that we can work only with lemmas of the parsed text of the single collection and doing so quickly prepare one or more resources of the type "Lexicon" to be used inside single collection. The disadvantage is that in each cycle we need again to browse through all pages of the scanned book in search for relevant lemmas and that takes a lot of time.

Supplementing at start

Another possible workflow is as follows: at first we create new contents file to be used by certain single collection. Then we don't fill the gaps with page numbers, but instead supplement the contents file with the lemmas found in original text of another collection. And we repeat this process again and again until all lemmas of the all original texts of the all relevant collections are included in the contents file. Then we browse through all pages of the scanned book, find the page numbers of the all lemmas and fill the gaps in contents file with these page numbers.

This workflow has the advantage that we must browse though all pages of the scanned book only once and this takes much less time than repeated browsing of the workflow described above. But the disadvantage is that we cannot work with the single collection at the time and must create huge contents file at once.

Cleaning page numbers

By using the first or the second workflow of those described above we get contents file of the resource of the type "Lexicon" which is being used by multiple collections. We may need to add another resource of the type "Lexicon" to the same collections. E.g. after having added lexicon by E. Robinson to every collection dedicated to certain book of the New Testament we may need to add F. Zorell's Lexicon Graecum Novi Testamenti to the same collections. In such cases we can shortcut the process by using the third workflow.

The contents file for the new resource of the type "Lexicon" should contain the same lemmas. This means that we can reuse existing contents file: we only need to remove page numbers, to change the name of the contents object and to change the name of the file. To remove page numbers from the contents file use Lexicon page numbers cleaner. Copy contents object from the contents file, paste it in the field "Lexicon contents object" and click the button "Clean page numbers":

Then save the outputted contents object without page numbers using relevant variable name and file name.

Importing after cleaning

It may happen that one resource of the type "Lexicon" has been added to multiple collections and another resource of the type "Lexicon" has been added to just single or several collections. E.g. such resource as E. Robinson's Greek and English lexicon of the New Testament has been added to every collection dedicated to certain book of the New Testament, but such resource as F. Zorell's Lexicon Graecum Novi Testamenti has been added to one or couple of collections. Now we need to add this resource to remaining collections.

In this case we need to use the following workflow. At first we should take that bigger contents file which contains comprehensive list of lemmas found in multiple collections and remove the page numbers from it. After that we need to import data into cleaned contents object from the contents object which contains less comprehensive list of lemmas from single collection or couple of collections. The value of the field "Lexicon contents object to be imported" should be contents object with all page numbers intact which contains less comprehensive list of lemmas from single collection or couple of collections.

The following conventions are used to format lexicon articles.

(1) On the top of the series of articles on words starting from certain letter is that letter wrapped by "h3" tag.

(2) Series of articles on words starting from certain letter is wrapped by "dl" tag.

(3) The words which the lexicon explains are wrapped by "dt" tag.

(4) The lemmas of explained words are wrapped by "span" tags having "class" attribute value "lexicon-heading". These "span" tags have "data-lemma" attribute. The value of this attribute is lowercase lemma of the explained word.

(5) Explanation of the word is wrapped by "dd" tag. Certain word may have several explanations, so below "dt" tag may stand several "dd" tags.

(6) Inner links, i.e. the links that lead from one place in the lexicon to another place in the same lexicon are wrapped by "a" tags. Inner link anchor has "class" attribute value "pc-anchor" and "id" attribute value consisting from lowercase lemma prefixed by certain string indicating lexicon title. Inner link himself has "class" attribute value "inner-link" and "data-anchor" attribute value consisting from lowercase lemma prefixed by certain string indicating lexicon title. In code example above there is link <a class="inner-link" data-anchor="green-nt-lex-βαρος" >βάρος</a> which leads to its anchor <a class="pc-anchor" id="green-nt-lex-βαρος" >Βάρος</a>.

Using these conventions lexicon articles should be formatted. There are no definite conventions how to format introduction, table of contents, various indexes, etc. Ask about it senior editor.

Coding policy

Dealing with alternative characters

In unicode two or more code points may encode characters which look identically or have only stylistic difference. In order for all functionality to work correctly the uniformity of encoding must be enforced by using only characters from certain "standard" set. To achieve this go to Unicode Characters changer and use the form to change unicode characters into "standard" ones.

Creating data file

Data file uses certain formatting conventions.

The file should contain "Grammars" heading and "References" heading both wrapped by "h2" tags.

Below the heading "Grammars" should be list of grammars. Each item of the list should be wrapped by "span" tag having class attribute "text_from". Every list item should contain abbreviated name of the grammar (usually author's surname) on the left and its bibliographical data to the right, separated by n-dash. Bibliographical data should be wrapped by "a" tag, having the value "_blank" of the attribute "target" and the relative URL as the value of the attribute "href".

Below the heading "References" should be list of references to grammars for the each line of the classical text. "span" tag having class attribute "line" should wrap all references for the single line. "span" tag having class attribute "line-number" should wrap line number of the line. The same "span" element should also have line number as the value of its "data-line-number" attribute.

Each reference to the grammar book consists of abbreviated grammar name and page number. Abbreviated name and page number should be wrapped by "a" tag having attribute "data-grammar" and attribute "data-page". The value of the attribute "data-grammar" should be the key of the relevant grammar book from the file _scanned_books.js. The value of the attribute "data-page" should be the number from the name of the image file (not the number of the page pictured!).

HTML code formatted using these tags should be converted to a string and made the value of javascript global variable whose name is the same as resource shortname. We listed grammar references resource whose resource shortname is "nt_titus_grammar_refs_by_parsedclassics", so this will be variable's name.

Creating resource of the type "diagram set" involves creating diagram images and data file.

Creating diagram images

To create diagram images Syntax diagram generator script should be used. This program generates syntax diagrams based on Modernized Reed-Kellogg diagram system. Syntax diagrams are generated from input in JSON format; input JSON can be generated by filling the form:

The value of the field "Parsed sentence code" should be the code from parsed text data file containing lines with the sentences to be diagrammed. Here we entered the code of the line 6:11 from apostle Paul's Epistle to Ephesians.

After entering the value of the field "Parsed sentence code" the button "Load sentence" should be clicked. It will result into two things: (a) above the splitter the sentence will be displayed, (b) inputs block "Syntactic relation" will be displayed below the heading "Syntax data":

Creating of syntax diagram is being done by manipulating inputs blocks such as that displayed after clicking the button "Load sentence". One line of inputs blocks describes one syntactic relation. The order of the lines is not essential.

There are three types of inputs blocks: (a) syntactic relation block; (b) word block; (c) phrase block.

Syntactic relation block stands first in the line of inputs blocks. The values of the fields of syntactic relation inputs block describe syntactic relation itself, i.e. in abstraction of words and/or phrases it relates. After the loading of the sentence new syntactic relation inputs block can be added by clicking the button "Syntactic relation". Syntactic relation inputs block can be removed by clicking the button "Remove". If we have two or more lines of inputs blocks, we can reorder them by dragging the handle ⇅ .

The values of the fields of word inputs block describe how the word functions inside syntactic relation. New word inputs block can be added by clicking the button "Word". Word inputs block can be removed by clicking the button "Remove" which is below the fields which describe the word.

The values of the fields of phrase inputs block describe how the phrase functions inside syntactic relation. New phrase inputs block can be added by clicking the button "Phrase". Phrase inputs block can be removed by clicking the button "Remove" which is below the fields which describe the phrase.

If we need to reorder word and/or phrase inputs blocks inside the line of inputs blocks, we can do it by dragging the handle ⇄ .

Only one Syntactic relation can be marked as "Root relation" by checking checkbox. (Root relation is that, which is displayed by the most left symbol in the diagram.)

Although the order of the lines of inputs blocks is not essential, it is recommended to analyze syntactic relations in certain order, which would allow to generate as often as possible partial diagram of the sentence. In order to be able to generate partial diagram as often as possible, we need:
(a) to start from root relation, if root relation relates only words, not phrases;
(b) if (root) relation relates some phrase, start analysis from that phrase and then analyze (root) relation;
(c) after analyzing (root) relation proceed with analyzing relations which relate words of (root) relation with other words;
(d) if word of (root) relation is related by some other relation to phrase, relation of that phrase should be analyzed first;
(e) use rules (b)-(d) for all non root relations in the structure of the sentence.

By using these simple rules we will be able to click the buttons "Generate JSON" and "Generate diagram" each time we have part of the sentence analyzed to its constituent words.

The words preceded by articles are really phrases formed by syntactic relation of specification, but since it would be very tedious to display this in diagrams, application of article to single word is displayed as if it were not phrase, but single word. Word form must be entered for each word inputs block using dropdown or manually.

In the field "Syntactic role" we entered "S." (subject) for the word "ὑμεῖς", "P." (predicator) for the word "ἐνδύσασθε" and "O." (object) for the word "τὴν πανοπλίαν". This field is optional; we need to be careful not to forget to enter syntactic role where needed. Entering syntactic role is needed when syntactic relation allows for the words it relates to have different syntactic roles and syntactic roles are not indicated in the diagram by the sign of certain syntactic relation.

In the field "Internal index" the value for the word "ἐνδύσασθε" was entered automatically when we selected word form using dropdown. As internal index can serve any unique string, but usually the string "wi" with padded number is added for implicit word and "wa" with padded number is added for word preceded by article. The field "Internal index" is required; its values are used to identify the words in different inputs blocks as the same.

The word "ὑμεῖς" has "(1)" as the value of the field "Subscript". Scanned resources

Scanning a book

The book needed for scanned resource should be scanned in full (pages from the title page to the last, not including covers and promotional materials). Each page should be scanned into separate JPG file. Scanning resolution should be at least 400dpi, scanning mode should be "color" or "greyscale".

Quite often it is more convenient to scan two pages of the book into one JPG file. In this case you will need to split each file into two files. This can be done usinge "TileMage" program (Win, free). This program places each pair of new files created into separate directory. So you will need to flatten directory structure. This can be made using program "Suction" (Win, free).

Quite often you may find a PDF file displaying scanned book. If its quality is good enough, PDF file can be used as the source of images for scanned resource. Unfortunately, quite often each page of PDF file is watermarked with some image or text, such as "Created with Cool PDF Soft". Such watermarks should be removed before converting PDF file to series of JPG images.

To remove watermark you can use Foxit Advanced PDF Editor (Win, commercial). Open watermarked PDF file, click "Object tool" in the toolbar and then click the watermark to select it:

Then in the menubar go Edit > Delete Across Pages... and in the Dialogue select "All Pages" and click the button "Delete". The watermark will be deleted from all pages:

Bulk download of images

At first you need to generate list of URL addresses of the image files to be downloaded. To do that go to URL list generator, fill the form and click "Generate URL list" button:

The value of the field "Start number" should be the number from the URL of the first page of the scanned book.

The value of the field "End number" should be the number from the URL of the last page of the scanned book.

The value of the field "Zero padding to length" should be the length of the numbers in the file names. E.g. if the file names of the scanned images are scanned-page00001.jpg, scanned-page00002.jpg, scanned-page00003.jpg etc., then the value of this field should be "5", because the numbers 00001, 00002, 00003 atc. contain 5 positions.

The value of the field "URL template" should be URL of the scanned image with the number changed into "{index}" variable.

Using regex

While editing code of the file of the transcribed resource, there is often the need to remove or replace certain parts of the HTML code. For example, you may need to remove "id" or "href" attributes, to remove opening or closing part of certain tags, to reverse order of tags, to extract certain numbers and put them in another place. The most efficient way to do this is using regular expressions (regex).

Regular expressions can be used with such code editors as Notepad++ (Win, free), ActiveState Komodo Edit (Win/Mac, free) and many others.

For using regular expressions with Notepad++ code editor go Search > Replace and inside the dialogue check "Regular expression", enter regex in the "Find" field, enter another expression in the field "Replace" or leave it blank if you need to delete and click "Replace All". Or click "Replace All in All Opened Documents" if you need to make replacements in several files. All strings which match the regular expression will be replaced:

For using regular expressions with ActiveState Komodo Edit code editor go Menu > Replace and inside the dialogue click the button "Treat the pattern as a regular expression", enter regex in the "Find" field, enter another expression in the field "Replace" or leave it blank if you need to delete and click "Replace all". All strings which match the regular expression will be deleted:

Appendix I. URL stucture

Almost everything you do in Library app will be reflected in URL. In other words, URL reflects the state of the Library app.

Only what is displayed in bottom subpane, right subpane, top right subpane, bottom right subpane is not reflected in URL.

URL is structured in the way that makes it easily readable.

To see how the state of the Library app is reflected in the URL let's analyze this example URL:

https://www.parsedclassics.com/classics.html#{"L":{"a":[["nt_titus|nt_titus_parsed_text","nt_titus|nt_titus_text_ed_robinson_pierpont"]],"b":[["nt_titus|nt_titus_tr_LA_s_hieronymus","nt_titus|nt_titus_tr_EN_young"],["nt_titus|greek_lexicon_to_nt_by_green"]]},"P":{"nt_titus":{"line":"2:8","word":"ἐντρέπω"}},"D":{"a":[["zwdfzv",50],["kke9uh",100,["i4yoe6","o1sscd"],0]],"b":[["z11wu3",50],["xewahx",50,["vowggx","c8neaw"],1,"max"],["1agxr3",50,["4ryhqx"],0]]}}

In this URL classics.html is the page of Library app, and the part after # is JSON object representing the state of the Library app. This JSON object has 3 keys – "L", "P" and "D", whose values are JSON objects.

"L" part of URL

Let's repeat "L" part of the URL:

"L":{"a":[["nt_titus|nt_titus_parsed_text","nt_titus|nt_titus_text_ed_robinson_pierpont"]],"b":[["nt_titus|nt_titus_tr_LA_s_hieronymus","nt_titus|nt_titus_tr_EN_young"],["nt_titus|greek_lexicon_to_nt_by_green"]]}

The key "L" means "layout". JSON object which is the value of the key "L" represents how the layout of the Library app is split into sections and panes and which resources are loaded.

The lower Latin keys "a" and "b" are indexes for the first and second section. If there were third, fouth etc. section, their keys would be "c", "d" etc.

Each lower Latin key has at its value two-level array, i.e. array whose members are arrays. The members of the first-level arrays are strings. Second-level array represents section, first level arrays represent panes and strings represent resources.

Inside the second-level array which is the value of the "a" key there is only one first-level array. It means that only one pane is inside the first section. Inside this first-level array there are two strings as its members. That means that the tabs of the only pane contain two resources. Each string representing resource contain two parts separated by pipe symbol: collection shortname and resource shortname. So, the first string represents resource whose shortname is "nt_titus_parsed_text" from collection whose shortname is "nt_titus"; the second string represents resource whose shortname is "nt_titus_text_ed_robinson_pierpont" from collection whose shortname is "nt_titus".

Inside the second-level array which is the value of the "b" key there are two first-level arrays. It means that there are two panes inside the second section. Inside the first of these two first-level arrays there are two strings as its members. That means that the tabs of the upper pane contain two resources. The first string represents resource whose shortname is "nt_titus_tr_LA_s_hieronymus" from collection whose shortname is "nt_titus"; the second string represents resource whose shortname is "nt_titus_tr_EN_young" from collection whose shortname is "nt_titus". Inside the second of two first-level arrays there is one string as its member. This string represents resource whose shortname is "greek_lexicon_to_nt_by_green" from collection whose shortname is "nt_titus".

"P" part of URL

Let's repeat "P" part of the URL:

"P":{"nt_titus":{"line":"2:8","word":"ἐντρέπω"}}

The key "P" means "position". JSON object which is the value of the key "P" represents at which position are opened resources of certain collection or collections.

Resources can be line-indexed (resources of the types Parsed text, Original text, Translation, Commentary, Commentary references, Grammar references, Diagram set, Audio recording) or word-indexed (resources of the types Concordance, Lexicon). Position of the line-indexed resources is represented by the value of the key "line"; position of the word-indexed resources is represented by the value of the key "word".

The meaning of JSON object in our example is as follows: inside the collection whose shortname is "nt_titus" line-indexed resources are opened at the line "2:8" and word-indexed resources are opened at the word "ἐντρέπω".

For representing position in word-indexed resources 2 optional keys can be used: "lexicon" and "lexentry". They are used in those cases when more than one word has the same lemma.

Let's inspect as an example such "P" part of the URL:

"P":{"nt_matthew":{"line":"9:38","word":"δέω","lexicon":"greek_lexicon_to_nt_by_green","lexentry":"2"}}

The meaning of JSON object in example is as follows: inside the collection whose shortname is "nt_matthew" line-indexed resources are opened at the line "2:8" and word-indexed resources are opened at the word "δέω"; the word-indexed resource whose shortname is "greek_lexicon_to_nt_by_green" should be opened at the second entry indexed by the word "δέω".

In case there are two or more collections selected in different tabs, the tabs in which the one collection is selected will be marked by circle colored in one color, the tabs in which the other collection is selected will be marked by circle colored in another color, etc. The color of the circle mark is indicated in URL by the value of the key "tag". Let's inspect as an example such "P" part of the URL:

"P":{"nt_luke":{"line":"1:15","tag":"green"},"nt_mark":{"line":"3:18","tag":"red"}}

The meaning of JSON object in example is as follows: inside the collection whose shortname is "nt_luke" line-indexed resources are opened at the line "1:15" and all tabs displaying resources belonging to this collection are marked by green circle mark; inside the collection whose shortname is "nt_mark" line-indexed resources are opened at the line "3:18" and all tabs displaying resources belonging to this collection are marked by red circle mark.

"D" part of URL

Let's repeat "D" part of the URL:

"D":{"a":[["zwdfzv",50],["kke9uh",100,["i4yoe6","o1sscd"],0]],"b":[["z11wu3",50],["xewahx",50,["vowggx","c8neaw"],1,"max"],["1agxr3",50,["4ryhqx"],0]]}

The key "D" means "dimensions". JSON object which is the value of the key "D" represents width of sections, height of panes, which tabs are active, which tab is maximized.

The lower Latin keys "a" and "b" are indexes for the first and second section. These are exactly the same keys as in part "L" of the URL.

Each lower Latin key has at its value second-level array, i.e. array whose members are arrays.

The key "a" has as its value second-level array which has two first-level arrays as its members.

The first of them is ["zwdfzv",50]. It represents section. "zwdfzv" is section's unique id, 50 is section''s width in percents.

The second of them is ["kke9uh",100,["i4yoe6","o1sscd"],0]]. It represents the only pane of the section. "kke9uh" is pane's unique id, 100 is pane's height in percents, ["i4yoe6","o1sscd"] is array of the unique ids of the pane's tabs, 0 is the index of the active tab.

The key "b" has as its value second-level array which has three first-level arrays as its members.

The first of them is ["z11wu3",50]. It represents section. "z11wu3" is section's unique id, 50 is section''s width in percents.

The second of them is ["xewahx",50,["vowggx","c8neaw"],1,"max"]. It represents the top pane of the section. "xewahx" is pane's unique id, 50 is pane's height in percents, ["vowggx","c8neaw"] is array of the unique ids of the pane's tabs, 1 is the index of the active tab, "max" signifies that the active tab is maximized.

The third of them is ["1agxr3",50,["4ryhqx"],0]. It represents the bottom pane of the section. "1agxr3" is pane's unique id, 50 is pane's height in percents, ["4ryhqx"] is array of the unique ids of the pane's tabs, 0 is the index of the active tab.

Appendix II. Platform architecture

Technology stack

Since classical texts and scholarship will be allways relevant, the design of the ParsedClassics website aimed at long term preservation of digital content. Long term preservation can be achieved by choosing widely used stack of technology compatible with broad range of both server and desktop environments. This led us to decision to choose HTML/CSS/Javascript stack, reject using any database software and server scripting languages and to use only relative URLs. As a result of this architectural decision ParsedClassics is static website which simply by copying can be "installed" on server or on desktop computer without any difference in behavior and used by pointing any modern browser.

Functions, sections, directories

ParsedClassics website is a platform performing these main functions:
(1) displaying digital editions of classical Greek and Latin texts;
(2) enabling creation of digital editions;
(3) documenting best practices of creating and using of digital editions;
(4) informing the lovers of classics about digital editions.

While developing ParsedClassics platform it was found that each of the four main functions is best performed if it's software is organized as separated unit, having its own directories, HTML, CSS and Javascript files. This way the platform is naturaly split into four sections:
(1) Library app section;
(2) Tools section;
(3) Docs section;
(4) Site section.

Such organization has following advantages: (a) changes in one section do not break anything in other sections; (b) making changes in one section do not force to upgrade other sections.
